# 開發者成長分析

這個 skill 透過分析您的 Claude Code 聊天互動並識別揭示優勢和成長領域的模式，提供有關您最近程式設計工作的個人化回饋。

## 何時使用此 Skill

當您想要：
- 了解最近工作中的開發模式和習慣
- 識別特定的技術差距或反覆出現的挑戰
- 發現哪些主題將受益於更深入的學習
- 獲得針對您實際工作模式量身定制的策劃學習資源
- 追蹤您最近專案的改進領域
- 尋找直接針對您正在開發的技能的高品質文章

此 skill 非常適合想要在無需等待程式碼審查的情況下獲得結構化成長回饋的開發者，以及偏好從自己工作歷史中獲得數據驅動見解的人。

## 此 Skill 的功能

這個 skill 對您的開發工作執行六步驟分析：

1. **讀取您的聊天歷史記錄**：存取過去 24-48 小時的本地 Claude Code 聊天歷史記錄，以了解您一直在做什麼。

2. **識別開發模式**：分析您正在解決的問題類型、使用的技術、遇到的挑戰，以及您如何處理不同類型的任務。

3. **檢測改進領域**：識別表明技能差距、反覆困難、低效方法或可能受益於更深入知識的領域的模式。

4. **生成個人化報告**：建立全面的報告，顯示您的工作摘要、識別的改進領域，以及成長的具體建議。

5. **尋找學習資源**：使用 HackerNews 策劃與您的改進領域直接相關的高品質文章和討論，為您提供針對實際開發工作量身定制的閱讀清單。

6. **傳送至您的 Slack 私訊**：自動將完整報告傳送至您自己的 Slack 直接訊息，以便您隨時隨地參考。

## 使用方法

要求 Claude 分析您最近的程式設計工作：

```
分析我最近聊天中的開發者成長
```

或更具體地說明哪個時間段：

```
分析我今天的工作並建議改進領域
```

此 skill 將生成格式化報告，包含：
- 您最近工作的概述
- 識別的關鍵改進領域
- 每個領域的具體建議
- 從 HackerNews 策劃的學習資源
- 您可以專注的行動項目

## 指示說明

當使用者請求分析他們最近工作的開發者成長或程式設計模式時：

1. **存取聊天歷史記錄**

   從 `~/.claude/history.jsonl` 讀取聊天歷史記錄。此檔案是 JSONL 格式，其中每行包含：
   - `display`：使用者的訊息/請求
   - `project`：正在處理的專案
   - `timestamp`：Unix 時間戳記（以毫秒為單位）
   - `pastedContents`：任何貼上的程式碼或內容

   根據目前時間戳記過濾過去 24-48 小時的條目。

2. **分析工作模式**

   從過濾的聊天中提取並分析以下內容：
   - **專案和領域**：使用者在處理什麼類型的專案？（例如後端、前端、DevOps、數據等）
   - **使用的技術**：對話中出現哪些語言、框架和工具？
   - **問題類型**：正在解決哪些類別的問題？（例如效能優化、除錯、功能實作、重構、設定/配置）
   - **遇到的挑戰**：使用者在哪些問題上掙扎？尋找：
     - 關於類似主題的重複問題
     - 需要多次嘗試才能解決的問題
     - 表明知識差距的問題
     - 複雜的架構決策
   - **方法模式**：使用者如何解決問題？（例如有條理、探索性、實驗性）

3. **識別改進領域**

   根據分析，識別 3-5 個使用者可以改進的特定領域。這些應該是：
   - **具體的**（不是像「改進程式設計技能」這樣含糊的）
   - **基於證據的**（根植於實際聊天歷史記錄）
   - **可行的**（可以進行的實際改進）
   - **優先排序的**（最有影響力的優先）

   良好改進領域的範例：
   - 「進階 TypeScript 模式（泛型、工具類型、型別守衛）- 您在 [特定專案] 中與型別安全掙扎」
   - 「錯誤處理和驗證 - 我注意到您修補了幾個與缺少 null 檢查相關的錯誤」
   - 「Async/await 模式 - 您最近的工作顯示一些競態條件和時序問題」
   - 「資料庫查詢優化 - 您多次重寫相同的查詢」

4. **生成報告**

   建立具有此結構的全面報告：

   ```markdown
   # 您的開發者成長報告

   **報告期間**：[昨天 / 今天 / [自訂日期範圍]]
   **最後更新**：[目前日期和時間]

   ## 工作摘要

   [2-3 段總結使用者所做的工作、涉及的專案、使用的技術和整體焦點領域]

   範例：
   「在過去 24 小時內，您主要專注於後端開發，涉及三個不同的專案。您的工作涉及 TypeScript、React 和部署基礎架構。您處理了功能實作、除錯和架構決策的混合，特別專注於 API 設計和資料庫優化。」

   ## 改進領域（優先排序）

   ### 1. [領域名稱]

   **為何重要**：[解釋為何這項技能對使用者的工作很重要]

   **我觀察到的**：[來自聊天歷史記錄的具體證據顯示此差距]

   **建議**：[在此領域改進的具體步驟]

   **技能提升時間**：[所需努力的簡要估計]

   ---

   [為另外 2-4 個領域重複]

   ## 觀察到的優勢

   [2-3 個要點突出您做得好的事情 - 應該繼續做的事情]

   ## 行動項目

   優先順序：
   1. [從最高優先級改進領域衍生的行動項目]
   2. [從下一個領域的行動項目]
   3. [從下一個領域的行動項目]

   ## 學習資源

   [將在下一步填充]
   ```

5. **搜尋學習資源**

   使用 Rube MCP 搜尋 HackerNews 上與每個改進領域相關的文章：

   - 對於每個改進領域，建構針對高品質資源的搜尋查詢
   - 使用 RUBE_SEARCH_TOOLS 搜尋 HackerNews，查詢如：
     - 「學習 [技術/模式] 最佳實踐」
     - 「[技術] 進階模式和技術」
     - 「在 [語言] 中除錯 [特定問題類型]」
   - 優先考慮高參與度的貼文（評論、按讚）
   - 對於每個領域，包含 2-3 篇最相關的文章，包括：
     - 文章標題
     - 發布日期
     - 為何相關的簡要描述
     - 文章連結

   將此段落新增至報告：

   ```markdown
   ## 策劃的學習資源

   ### 針對：[改進領域]

   1. **[文章標題]** - [日期]
      [描述它涵蓋的內容以及為何與您的改進領域相關]
      [連結]

   2. **[文章標題]** - [日期]
      [描述]
      [連結]

   [為其他改進領域重複]
   ```

6. **呈現完整報告**

   以乾淨、可讀的格式提供報告，使用者可以：
   - 快速掃描關鍵要點
   - 用於專注的學習計劃
   - 在接下來的一週內參考，因為他們致力於改進
   - 如果他們想要外部回饋，可以與導師分享

7. **將報告傳送至 Slack 私訊**

   使用 Rube MCP 將完整報告傳送至使用者自己的 Slack 私訊：

   - 透過 RUBE_SEARCH_TOOLS 檢查 Slack 連接是否活躍
   - 如果未連接，使用 RUBE_MANAGE_CONNECTIONS 啟動 Slack 驗證
   - 使用 RUBE_MULTI_EXECUTE_TOOL 將報告作為格式化訊息傳送：
     - 將報告標題和期間作為第一條訊息傳送
     - 將報告分解為邏輯段落（摘要、改進、優勢、行動、資源）
     - 將每個段落格式化為結構良好的 Slack 訊息，並使用適當的 markdown
     - 為學習資源包含可點擊的連結
   - 在 CLI 輸出中確認傳送

   這確保使用者在他們定期檢查的地方擁有報告，並可以在整週內參考它。

## 使用範例

### 輸出

```markdown
# 您的開發者成長報告

**報告期間**：2024 年 11 月 9-10 日
**最後更新**：2024 年 11 月 10 日，UTC 晚上 9:15

## 工作摘要

在過去兩天內，您專注於後端基礎架構和 API 開發。您的主要專案是一個開源展示應用程式，在連接管理、UI 改進和部署配置方面取得了重大進展。您使用 TypeScript、React 和 Node.js 工作，處理從數據安全到響應式設計的挑戰。您的工作在實作功能和解決技術債務之間取得平衡。

## 改進領域（優先排序）

### 1. 進階 TypeScript 模式和型別安全

**為何重要**：TypeScript 是您工作的核心，但利用其進階功能（泛型、工具類型、條件類型、型別守衛）可以顯著提高程式碼可靠性並減少執行時錯誤。更好的型別安全在編譯時而非生產環境中捕獲錯誤。

**我觀察到的**：在您最近的聊天中，您正在處理連接資料結構，並在正確輸入驗證配置時掙扎了幾次。您還必須迭代不同連接狀態的聯合類型。有機會更有效地使用可辨識聯合和型別守衛。

**建議**：學習 TypeScript 的進階型別系統，特別是工具類型（Omit、Pick、Record）、條件類型和可辨識聯合。將這些模式應用於您的連接配置處理和驗證狀態管理。

**技能提升時間**：5-8 小時的專注學習和練習

### 2. UI 中的安全資料處理和資訊隱藏

**為何重要**：您識別並修復了敏感連接資料在控制台中顯示的安全問題。防止資訊洩漏對於處理使用者憑證和 API 金鑰的應用程式至關重要。這裡的良好實踐可以防止安全事件和使用者信任違規。

**我觀察到的**：您發現「您的應用程式」頁面顯示完整的連接資料，包括驗證配置。這顯示出良好的安全本能，下一步是將其建構到您的預設思維中，當處理敏感資訊時。

**建議**：審查前端應用程式中處理敏感資料的安全最佳實踐。建立可重複使用的模式，在顯示敏感資訊之前進行過濾/遮罩。考慮實作一個安全資料層，明確將可在 UI 中顯示的內容列入白名單。

**技能提升時間**：3-4 小時

### 3. 元件架構和響應式 UI 模式

**為何重要**：您正在設計需要在不同螢幕尺寸和使用者互動中運作的 UI。強大的元件架構使建構複雜 UI 更容易而不會出現錯誤，並提高可維護性。

**我觀察到的**：您處理了「市集」UI（以前是瀏覽工具），從設計圖像重新建立它。您還識別並修復了內容溢出容器的捲動問題。有機會加強您對版面配置包含和響應式設計模式的理解。

**建議**：學習 React 元件組合模式和 CSS 版面配置最佳實踐（特別是 flexbox 和 grid）。專注於容器查詢和防止溢出問題的響應式模式。研究元件組合庫和設計系統方法。

**技能提升時間**：6-10 小時（取決於深度）

## 觀察到的優勢

- **安全意識**：您在問題變成實際問題之前主動識別資料洩漏問題
- **迭代精煉**：您有條理地處理 UI 需求，提出澄清問題並改進設計
- **全端能力**：您可以輕鬆地跨後端 API、前端 UI 和部署問題工作
- **問題解決方法**：您將複雜任務分解為可管理的步驟

## 行動項目

優先順序：
1. 花 1-2 小時學習 TypeScript 工具類型和可辨識聯合；應用於您的連接資料結構
2. 為您的專案記錄安全模式（哪些資料可以安全顯示、過濾/遮罩功能）
3. 學習一篇關於進階 React 模式的文章，並將一種模式應用於您目前的 UI 工作
4. 設定專注於型別安全和資料安全的程式碼審查檢查清單，用於未來的 PR
```

## 技巧和最佳實踐

- 每週執行一次此分析，以追蹤您隨時間的改進軌跡
- 一次選擇一個改進領域，並在幾天內專注於它，然後再進入下一個
- 將學習資源用作學習指南；完成推薦的材料並練習應用模式
- 在專注於某個領域一週後重新訪問此報告，以查看您的工作模式如何變化
- 學習資源是特意為您的實際工作策劃的，而非通用主題，因此它們將與您正在建構的內容高度相關

## 如何維持準確性和品質

此 skill：
- 從時間戳記的聊天歷史記錄中分析您的實際工作模式
- 生成根植於真實專案的基於證據的建議
- 策劃直接針對您識別的差距的學習資源
- 專注於可行的改進，而非含糊的回饋
- 根據複雜性提供具體的時間估計
- 優先考慮將對您的開發速度產生最大影響的領域
