# 內容研究寫作助手

這個 skill 作為您的寫作夥伴，協助您研究、大綱規劃、草稿撰寫和內容精煉，同時保持您獨特的語調和風格。

## 何時使用此 Skill

- 撰寫部落格文章、文章或電子報
- 創作教育內容或教學
- 起草思想領袖文章
- 研究和撰寫案例研究
- 製作附有來源的技術文件
- 撰寫帶有適當引用和參考文獻的內容
- 改善開頭和引言
- 在寫作過程中獲得逐段回饋

## 此 Skill 的功能

1. **協作式大綱規劃**：協助您將想法組織成連貫的大綱
2. **研究協助**：尋找相關資訊並新增引用
3. **開頭改善**：加強您的開場白以吸引注意力
4. **段落回饋**：在您撰寫時審查每個段落
5. **語調保留**：維持您的寫作風格和語調
6. **引用管理**：正確新增和格式化參考文獻
7. **迭代精煉**：透過多次草稿協助您改進

## 使用方法

### 設定您的寫作環境

為您的文章建立專屬資料夾：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

建立您的草稿檔案：
```
touch article-draft.md
```

從這個目錄開啟 Claude Code 並開始寫作。

### 基本工作流程

1. **從大綱開始**：
```
幫我為關於 [主題] 的文章建立大綱
```

2. **研究並新增引用**：
```
研究 [特定主題] 並在我的大綱中新增引用
```

3. **改善開頭**：
```
這是我的引言。幫我讓開頭更引人注目。
```

4. **獲得段落回饋**：
```
我剛完成「為何重要」段落。審查它並給予回饋。
```

5. **精煉和潤飾**：
```
審查完整草稿的流暢度、清晰度和一致性。
```

## 指示說明

當使用者請求寫作協助時：

1. **了解寫作專案**

   詢問澄清問題：
   - 主題和主要論點是什麼？
   - 目標讀者是誰？
   - 期望的長度/格式是什麼？
   - 您的目標是什麼？（教育、說服、娛樂、解釋）
   - 是否有現有的研究或來源要納入？
   - 您的寫作風格是什麼？（正式、對話式、技術性）

2. **協作式大綱規劃**

   協助組織內容結構：

   ```markdown
   # 文章大綱：[標題]

   ## 開頭
   - [開場句子/故事/統計數據]
   - [為何讀者應該關心]

   ## 引言
   - 背景和脈絡
   - 問題陳述
   - 本文涵蓋內容

   ## 主要段落

   ### 段落 1：[標題]
   - 要點 A
   - 要點 B
   - 範例/證據
   - [需要研究：特定主題]

   ### 段落 2：[標題]
   - 要點 C
   - 要點 D
   - 需要數據/引用

   ### 段落 3：[標題]
   - 要點 E
   - 反論點
   - 解決方案

   ## 結論
   - 主要觀點摘要
   - 行動呼籲
   - 最終想法

   ## 研究待辦事項
   - [ ] 尋找關於 [主題] 的數據
   - [ ] 獲取 [概念] 的範例
   - [ ] 為 [主張] 尋找引用來源
   ```

   **迭代大綱**：
   - 根據回饋調整
   - 確保邏輯流程
   - 識別研究缺口
   - 標記需要深入探討的段落

3. **進行研究**

   當使用者請求某個主題的研究時：

   - 搜尋相關資訊
   - 尋找可信來源
   - 提取關鍵事實、引言和數據
   - 以要求的格式新增引用

   範例輸出：
   ```markdown
   ## 研究：AI 對生產力的影響

   主要發現：

   1. **生產力提升**：研究顯示內容創作任務節省 40% 時間 [1]

   2. **採用率**：67% 的知識工作者每週使用 AI 工具 [2]

   3. **專家引言**：「AI 增強而非取代人類創造力」- Jane Smith 博士，MIT [3]

   引用文獻：
   [1] McKinsey Global Institute. (2024). 「生成式 AI 的經濟潛力」
   [2] Stack Overflow 開發者調查 (2024)
   [3] Smith, J. (2024). MIT Technology Review 訪談

   已新增至大綱的段落 2。
   ```

4. **改善開頭**

   當使用者分享引言時，分析並加強：

   **目前開頭分析**：
   - 有效之處：[正面元素]
   - 可以更強的地方：[改進領域]
   - 情感影響：[目前 vs. 潛在]

   **建議替代方案**：

   選項 1：[大膽陳述]
   > [範例]
   *為何有效：[解釋]*

   選項 2：[個人故事]
   > [範例]
   *為何有效：[解釋]*

   選項 3：[令人驚訝的數據]
   > [範例]
   *為何有效：[解釋]*

   **開頭檢查問題**：
   - 是否創造好奇心？
   - 是否承諾價值？
   - 是否足夠具體？
   - 是否符合讀者需求？

5. **提供逐段回饋**

   當使用者撰寫每個段落時，審查以下方面：

   ```markdown
   # 回饋：[段落名稱]

   ## 做得好 ✓
   - [優點 1]
   - [優點 2]
   - [優點 3]

   ## 改進建議

   ### 清晰度
   - [特定問題] → [建議修正]
   - [複雜句子] → [更簡單的替代方案]

   ### 流暢度
   - [轉折問題] → [更好的連接]
   - [段落順序] → [建議重新排序]

   ### 證據
   - [需要支持的主張] → [新增引用或範例]
   - [泛泛陳述] → [讓它更具體]

   ### 風格
   - [語調不一致] → [更符合您的語調]
   - [用詞選擇] → [更強的替代方案]

   ## 具體行編輯

   原文：
   > [從草稿中精確引用]

   建議：
   > [改進版本]

   原因：[解釋]

   ## 值得思考的問題
   - [發人深省的問題 1]
   - [發人深省的問題 2]

   準備進入下一段落！
   ```

6. **保留作者語調**

   重要原則：

   - **了解他們的風格**：閱讀現有的寫作樣本
   - **建議，不要替代**：提供選項，而非指令
   - **符合語調**：正式、隨意、技術性、友善
   - **尊重選擇**：如果他們偏好自己的版本，支持它
   - **增強，不要覆蓋**：讓他們的寫作更好，而非不同

   定期詢問：
   - 「這聽起來像您嗎？」
   - 「這是正確的語調嗎？」
   - 「我應該更/較不[正式/隨意/技術性]嗎？」

7. **引用管理**

   根據使用者偏好處理參考文獻：

   **內嵌引用**：
   ```markdown
   研究顯示生產力提升 40%（McKinsey, 2024）。
   ```

   **編號參考**：
   ```markdown
   研究顯示生產力提升 40% [1]。

   [1] McKinsey Global Institute. (2024)...
   ```

   **註腳風格**：
   ```markdown
   研究顯示生產力提升 40%^1

   ^1: McKinsey Global Institute. (2024)...
   ```

   維護引用清單：
   ```markdown
   ## 參考文獻

   1. 作者. (年份). 「標題」. 出版物.
   2. 作者. (年份). 「標題」. 出版物.
   ...
   ```

8. **最終審查和潤飾**

   當草稿完成時，提供全面回饋：

   ```markdown
   # 完整草稿審查

   ## 整體評估

   **優勢**：
   - [主要優勢 1]
   - [主要優勢 2]
   - [主要優勢 3]

   **影響力**：[整體效果評估]

   ## 結構與流暢度
   - [組織評論]
   - [轉折品質]
   - [節奏評估]

   ## 內容品質
   - [論點強度]
   - [證據充分性]
   - [範例效果]

   ## 技術品質
   - 文法和機制：[評估]
   - 一致性：[評估]
   - 引用：[完整性檢查]

   ## 可讀性
   - 清晰度評分：[評估]
   - 句子多樣性：[評估]
   - 段落長度：[評估]

   ## 最終潤飾建議

   1. **引言**：[具體改進]
   2. **正文**：[具體改進]
   3. **結論**：[具體改進]
   4. **標題**：[如需要的選項]

   ## 發布前檢查清單
   - [ ] 所有主張都有來源
   - [ ] 引用格式化
   - [ ] 範例清晰
   - [ ] 轉折流暢
   - [ ] 行動呼籲存在
   - [ ] 校對錯字

   準備發布！🚀
   ```

## 範例

### 範例 1：Teresa Torres 的工作流程

**使用者**：「我正在撰寫關於持續探索的文章。幫我建立大綱。」

**流程**：
1. 協作大綱結構
2. 識別研究需求
3. 使用者開始撰寫引言
4. 審查並改善開頭
5. 使用者撰寫每個段落
6. 在每個段落後提供回饋
7. 進行研究並新增引用
8. 最終審查完整草稿
9. 潤飾並準備發布

**結果**：研究充分、引用正確的文章，以 Teresa 的語調撰寫，具有強大的結構和流暢度。

### 範例 2：研究密集型文章

**使用者**：「我正在撰寫關於 AI 對產品管理影響的文章。幫我找到當前數據和範例。」

**輸出**：
```markdown
## 彙整的研究

### 最新數據
- Gartner：到 2025 年，80% 的 PM 將使用 AI 工具 [1]
- 調查：45% 報告更快的功能發布速度 [2]
- 案例研究：Airbnb 的 AI 輔助優先順序排序 [3]

### 專家引言
- 「AI 放大 PM 判斷力，而非取代它」- Marty Cagan
- [附引用的其他引言]

### 真實範例
1. **公司 A**：使用 AI 進行使用者研究綜合
   - 結果：節省 60% 時間
   - 來源：[引用]

2. **公司 B**：AI 驅動的路線圖分析
   - 結果：更好的利害關係人對齊
   - 來源：[引用]

所有內容已新增至您的大綱並附上適當引用。
```

### 範例 3：開頭改善

**使用者的原始開頭**：
> 「產品管理因為 AI 而改變。在本文中，我將討論 AI 影響產品經理的一些方式。」

**改進選項**：

**選項 1（數據驅動）**：
> 「上個月，我讓 AI 分析 500 場客戶訪談。它花了 30 分鐘而非 3 週。產品管理將永遠不同。」

**選項 2（問題式）**：
> 「如果您可以在早晨咖啡前與每位客戶對話、閱讀每條評論、分析每張支援工單，會怎樣？」

**選項 3（故事式）**：
> 「Sarah 花了兩週打造錯誤的功能。不是因為她不了解使用者，而是因為她無法快速處理數百場訪談以發現模式。」

### 範例 4：段落回饋

**使用者**：「剛完成『常見錯誤』段落。有什麼想法？」

**回應**：
```markdown
# 回饋：常見錯誤段落

## 做得好 ✓
- 強大的範例（Slack 通知故事）
- 清晰的結構（3 個錯誤，乾淨的排版）
- 每個錯誤的實用建議

## 建議

### 讓它更具體
您的第二點說「糟糕的優先順序排序」，但可以更強：

目前：
> 「許多團隊優先順序排序不當，專注於功能而非成果。」

建議：
> 「我見過團隊在一季內發布 14 個功能，卻沒有推動任何關鍵指標。他們優先考慮活動而非進展。」

### 新增數據
第三個錯誤將受益於證據：
> 「[新增引用]：研究顯示，沒有定期使用者接觸的團隊打造未使用功能的可能性高 3 倍 [需要來源]」

### 流暢度改進
考慮重新排序：錯誤 3 → 錯誤 2 → 錯誤 1
這從小影響到大影響建構。

準備進入下一段落！
```

## 寫作工作流程

### 部落格文章工作流程
1. 一起規劃大綱
2. 研究關鍵點
3. 撰寫引言 → 獲得回饋
4. 撰寫正文段落 → 每段回饋
5. 撰寫結論 → 最終審查
6. 潤飾和編輯

### 電子報工作流程
1. 討論開頭想法
2. 快速大綱（較短格式）
3. 一次完成草稿
4. 審查清晰度和連結
5. 快速潤飾

### 技術教學工作流程
1. 大綱步驟
2. 撰寫程式碼範例
3. 新增解釋
4. 測試指示
5. 新增疑難排解段落
6. 最終審查準確性

### 思想領袖工作流程
1. 腦力激盪獨特角度
2. 研究現有觀點
3. 發展您的論點
4. 以強烈觀點撰寫
5. 新增支持證據
6. 撰寫引人入勝的結論

## 專業技巧

1. **在 VS Code 中工作**：比網頁版 Claude 更適合長篇寫作
2. **一次一段**：逐步獲得回饋
3. **分開儲存研究**：保留一個 research.md 檔案
4. **版本化您的草稿**：article-v1.md、article-v2.md 等
5. **朗讀**：使用回饋識別不順的句子
6. **設定截止日期**：「我想今天完成草稿」
7. **休息**：撰寫、獲得回饋、暫停、修改

```
~/writing/article-name/

└── sources/            # 參考資料
    ├── study1.pdf
    └── article2.md
```

## 最佳實踐

### 研究方面
- 引用前驗證來源
- 盡可能使用最新數據
- 平衡不同觀點
- 連結到原始來源

### 回饋方面
- 具體說明您想要什麼：「這太技術性了嗎？」
- 分享您的顧慮：「我擔心這段拖沓」
- 提出問題：「這在邏輯上流暢嗎？」
- 要求替代方案：「還有什麼其他方式可以解釋這個？」

### 語調方面
- 分享您的寫作範例
- 指定語調偏好
- 指出好的匹配：「那聽起來像我！」
- 標記不匹配：「對我的風格來說太正式了」
