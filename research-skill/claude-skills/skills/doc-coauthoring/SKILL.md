# 文档协作编写工作流程

此技能为引导用户完成协作文档创建提供结构化工作流程。作为主动的引导者，引导用户完成三个阶段：上下文收集、完善与结构、读者测试。

## 何时提供此工作流程

**触发条件：**
- 用户提到编写文档："写一份文档"、"起草一份提案"、"创建规范"、"撰写"
- 用户提到特定的文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始一项重要的写作任务

**初始提议：**
向用户提供协作编写文档的结构化工作流程。解释三个阶段：

1. **上下文收集**：用户提供所有相关上下文，同时 Claude 提出澄清问题
2. **完善与结构**：通过头脑风暴和编辑迭代构建每个部分
3. **读者测试**：使用全新的 Claude（无上下文）测试文档，在其他人在阅读之前发现盲点

解释这种方法有助于确保文档在其他人阅读时（包括当他们将其粘贴到 Claude 中时）能够有效工作。询问他们是否想尝试此工作流程，还是更喜欢自由形式的工作方式。

如果用户拒绝，则以自由形式工作。如果用户接受，则进入第 1 阶段。

## 第 1 阶段：上下文收集

**目标：**缩小用户所知内容与 Claude 所知内容之间的差距，以便在后续提供智能指导。

### 初始问题

首先询问用户关于文档的元上下文：

1. 这是什么类型的文档？（例如：技术规范、决策文档、提案）
2. 主要受众是谁？
3. 读者阅读后期望产生什么影响？
4. 是否有要遵循的模板或特定格式？
5. 还有其他约束或上下文需要了解吗？

告知他们可以用简写方式回答，或以最适合他们的方式转储信息。

**如果用户提供模板或提到文档类型：**
- 询问他们是否有要分享的模板文档
- 如果他们提供共享文档的链接，使用相应的集成来获取它
- 如果他们提供文件，则读取它

**如果用户提到编辑现有的共享文档：**
- 使用相应的集成来读取当前状态
- 检查是否有不带 alt-text 的图片
- 如果存在没有 alt-text 的图片，解释当其他人使用 Claude 理解文档时，Claude 将无法看到它们。询问他们是否想要生成 alt-text。如果需要，请求他们将每张图片粘贴到聊天中以便生成描述性 alt-text。

### 信息转储

初始问题回答后，鼓励用户转储他们拥有的所有上下文。请求信息，例如：
- 项目/问题的背景
- 相关的团队讨论或共享文档
- 为什么不使用其他解决方案
- 组织上下文（团队动态、过去的事件、政治）
- 时间压力或约束
- 技术架构或依赖关系
- 利益相关者的关切

建议他们不用担心组织 - 只需要把所有内容都输出。提供多种提供上下文的方式：
- 意识流式信息转储
- 指向要阅读的团队频道或线程
- 链接到共享文档

**如果集成可用**（例如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 服务器），提及这些可用于直接引入上下文。

**如果未检测到集成且在 Claude.ai 或 Claude 应用中：** 建议他们可以在 Claude 设置中启用连接器，以允许直接从消息应用程序和文档存储中提取上下文。

告知他们在完成初始转储后会提出澄清问题。

**在上下文收集期间：**

- 如果用户提到团队频道或共享文档：
  - 如果集成可用：告知他们内容现在将被读取，然后使用相应的集成
  - 如果集成不可用：解释缺乏访问权限。建议他们在 Claude 设置中启用连接器，或直接粘贴相关内容

- 如果用户提到未知实体/项目：
  - 询问是否应搜索连接的工具以了解更多信息
  - 在搜索之前等待用户确认

- 随着用户提供上下文，跟踪正在学习的内容以及仍然不清楚的内容

**提出澄清问题：**

当用户表示已完成初始转储时（或在提供了大量上下文后），提出澄清问题以确保理解：

根据上下文中的差距生成 5-10 个编号的问题。

告知他们可以使用简写来回答（例如："1：是，2：见 #channel，3：否因为向后兼容"），链接到更多文档，指向要阅读的频道，或继续信息转储。只要对他们来说最高效即可。

**退出条件：**
当问题显示理解时 - 即可以询问边缘情况和权衡而无需解释基础知识时 - 表明已收集了足够的上下文。

**过渡：**
询问他们在此阶段是否还有更多上下文要提供，或者是否可以进入起草文档阶段。

如果用户想要添加更多内容，则让他们添加。准备好后，进入第 2 阶段。

## 第 2 阶段：完善与结构

**目标：** 通过头脑风暴、策划和迭代完善逐节构建文档。

**给用户的指令：**
解释文档将逐节构建。对于每个部分：
1. 将提出关于应包含内容的澄清问题
2. 将头脑风暴 5-20 个选项
3. 用户将指示保留/删除/合并哪些内容
4. 将起草该部分
5. 将通过精细编辑进行完善

从最未知的部分开始（通常是核心决策/提案），然后处理其余部分。

**部分排序：**

如果文档结构清晰：
询问他们想从哪个部分开始。

建议从最未知的部分开始。对于决策文档，通常是核心提案。对于规范，通常是技术方法。摘要部分最好留到最后。

如果用户不知道需要哪些部分：
根据文档类型和模板，建议适合文档类型的 3-5 个部分。

询问此结构是否有效，或者他们是否想要调整。

**一旦结构达成一致：**

使用占位符文本为所有部分创建初始文档结构。

**如果可以访问工件：**
使用 `create_file` 创建工件。这为 Claude 和用户提供了一个可工作的脚手架。

告知他们将为所有部分创建带有占位符的初始结构。

创建包含所有部分标题和简短占位符文本（如"[待写入]"或"[此处内容]"）的工件。

提供脚手架链接并指示现在可以填写每个部分。

**如果无法访问工件：**
在工作目录中创建一个 markdown 文件。适当命名（例如 `decision-doc.md`、`technical-spec.md`）。

告知他们将为所有部分创建带有占位符的初始结构。

创建包含所有部分标题和占位符文本的文件。

确认文件名已创建，并指示现在可以填写每个部分。

**对于每个部分：**

### 步骤 1：澄清问题

宣布将在 [部分名称] 部分开始工作。提出 5-10 个关于应包含内容的澄清问题：

根据上下文和部分目的生成 5-10 个具体问题。

告知他们可以使用简写回答，或者只指出需要涵盖的重要内容。

### 步骤 2：头脑风暴

对于 [部分名称] 部分，头脑风暴 [5-20] 个可能包含的内容，具体取决于部分的复杂性。寻找：
- 可能已被遗忘的共享上下文
- 尚未提及的角度或考虑因素

根据部分复杂性生成 5-20 个编号选项。最后，如果他们想要其他选项，提议进行更多头脑风暴。

### 步骤 3：策划

询问应保留、删除或合并哪些点。请求简短理由以帮助了解下一部分的优先级。

提供示例：
- "保留 1、4、7、9"
- "删除 3（与 1 重复）"
- "删除 6（受众已经知道这一点）"
- "合并 11 和 12"

**如果用户给出自由形式反馈**（例如"看起来不错"或"我喜欢大部分但..."）而不是编号选择，则提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用它。

### 步骤 4：缺口检查

根据他们的选择，询问 [部分名称] 部分是否有任何重要遗漏。

### 步骤 5：起草

使用 `str_replace` 将此部分的占位符文本替换为实际起草的内容。

宣布现在将根据他们选择的内容起草 [部分名称] 部分。

**如果使用工件：**
起草后，提供工件链接。

要求他们通读并指示要更改的内容。注意，具体说明有助于为下一部分学习。

**如果使用文件（无工件）：**
起草后，确认完成。

告知 [部分名称] 部分已在 [文件名] 中起草。要求他们通读并指示要更改的内容。注意，具体说明有助于为下一部分学习。

**给用户的关键指令（在起草第一部分时包括）：**
提供说明：而不是直接编辑文档，要求他们指示要更改的内容。这有助于为未来的部分学习他们的风格。例如："删除 X 项目 - 已由 Y 覆盖"或"使第三段更简洁"。

### 步骤 6：迭代完善

随着用户提供反馈：
- 使用 `str_replace` 进行编辑（从不重印整个文档）
- **如果使用工件：** 在每次编辑后提供工件链接
- **如果使用文件：** 只确认编辑完成
- 如果用户直接编辑文档并要求读取：在心理上记录他们所做的更改，并在未来的部分中记住这些更改（这显示了他们的偏好）

**继续迭代**，直到用户对该部分满意。

### 质量检查

在连续 3 次迭代没有实质性更改后，询问是否可以在不丢失重要信息的情况下删除任何内容。

该部分完成后，确认 [部分名称] 完成。询问是否准备好进入下一部分。

**对所有部分重复此过程。**

### 接近完成

当接近完成时（完成 80% 以上的部分），宣布打算重新阅读整个文档并检查：
- 各部分之间的流程和一致性
- 冗余或矛盾
- 任何感觉像"草率"或通用填充物的内容
- 是否每句话都有分量

阅读整个文档并提供反馈。

**当所有部分都已起草和完善时：**
宣布所有部分已起草。表明打算再审查一次完整文档。

审查整体连贯性、流程、完整性。

提供任何最终建议。

询问是否准备好进入读者测试，或者他们是否想要完善其他内容。

## 第 3 阶段：读者测试

**目标：** 使用全新的 Claude（无上下文泄漏）测试文档，以验证其对读者有效。

**给用户的指令：**
解释现在将进行测试，以查看文档是否真正对读者有效。这会发现盲点 - 对作者有意义但可能会让其他人困惑的事情。

### 测试方法

**如果可以访问子代理（例如在 Claude Code 中）：**

直接执行测试，无需用户参与。

### 步骤 1：预测读者问题

宣布打算预测读者在尝试发现此文档时可能会问的问题。

生成 5-10 个读者实际会问的问题。

### 步骤 2：使用子代理测试

宣布这些问题将在全新的 Claude 实例（无来自此对话的上下文）中进行测试。

对于每个问题，调用仅包含文档内容和问题的子代理。

总结读者 Claude 在每个问题上的正确/错误之处。

### 步骤 3：运行额外检查

宣布将执行额外检查。

调用子代理以检查歧义、错误假设、矛盾。

总结发现的任何问题。

### 步骤 4：报告和修复

如果发现问题：
报告读者 Claude 在特定问题上遇到困难。

列出具体问题。

表明打算修复这些缺口。

循环回到有问题的部分进行完善。

---

**如果无法访问子代理（例如 claude.ai 网页界面）：**

用户需要手动进行测试。

### 步骤 1：预测读者问题

询问人们在尝试发现此文档时可能会问什么问题。他们会在 Claude.ai 中输入什么？

生成 5-10 个读者实际会问的问题。

### 步骤 2：设置测试

提供测试说明：
1. 打开一个新的 Claude 对话：https://claude.ai
2. 粘贴或共享文档内容（如果使用启用连接器的共享文档平台，则提供链接）
3. 向读者 Claude 询问生成的问题

对于每个问题，指示读者 Claude 提供：
- 答案
- 是否有任何歧义或不清楚的内容
- 文档假设读者已经知道哪些知识/上下文

检查读者 Claude 是否给出正确答案或误解任何内容。

### 步骤 3：额外检查

还向读者 Claude 询问：
- "此文档中哪些内容可能对读者来说是模棱两可或不清楚的？"
- "此文档假设读者已经具备哪些知识或上下文？"
- "是否有任何内部矛盾或不一致？"

### 步骤 4：根据结果迭代

询问读者 Claude 弄错或遇到困难的内容。表明打算修复那些缺口。

循环回到任何有问题的部分进行完善。

---

### 退出条件（两种方法）

当读者 Claude 一致地正确回答问题并且不再发现新的缺口或歧义时，文档就准备好了。

## 最终审查

当读者测试通过时：
宣布文档已通过读者 Claude 测试。完成之前：

1. 建议他们自己再做一次通读 - 他们拥有此文档并对其质量负责
2. 建议再次检查任何事实、链接或技术细节
3. 询问他们验证是否达到了他们想要的影响

询问他们是否想要最后一次审查，或者工作是否完成。

**如果用户想要最终审查，则提供审查。否则：**
宣布文档完成。提供一些最终提示：
- 考虑在附录中链接此对话，以便读者可以看到文档是如何开发的
- 使用附录提供深度而不膨胀主文档
- 根据真实读者的反馈更新文档

## 有效指导的提示

**语气：**
- 直接和程序化
- 当影响用户行为时简要解释理由
- 不要试图"推销"方法 - 只是执行它

**处理偏差：**
- 如果用户想跳过阶段：询问他们是否想跳过此阶段并以自由形式编写
- 如果用户似乎感到沮丧：承认这比预期花费的时间更长。建议加快速度的方法
- 始终赋予用户调整流程的权限

**上下文管理：**
- 在整个过程中，如果缺少关于所提及内容的上下文，则主动询问
- 不要让缺口累积 - 在出现时解决它们

**工件管理：**
- 使用 `create_file` 起草完整部分
- 使用 `str_replace` 进行所有编辑
- 在每次更改后提供工件链接
- 从不将工件用于头脑风暴列表 - 这只是对话

**质量胜于速度：**
- 不要匆忙完成阶段
- 每次迭代都应进行有意义的改进
- 目标是一个真正对读者有效的文档
